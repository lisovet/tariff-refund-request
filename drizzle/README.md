# Drizzle migrations

This directory holds generated SQL migrations. Generated by `npm run db:generate`, applied by `npm run db:migrate`.

## Workflow

1. Add or modify a table in `src/shared/infra/db/schema.ts` (and the per-context schema files registered there).
2. Run `npm run db:generate` — drizzle-kit emits a numbered SQL file here.
3. Review the SQL diff carefully (especially destructive changes).
4. Commit the migration alongside the schema change.
5. Apply with `npm run db:migrate` (CI does this automatically; locally against your dev environment).

## Per-environment strategy (per ADR 003)

| Environment | Database |
| --- | --- |
| Local dev | Railway dev Postgres (one per developer) — `DATABASE_URL` in `.env.local` |
| PR preview | Railway preview environment per PR |
| Staging | Railway `staging` environment |
| Production | Railway `production` environment (swappable to AWS RDS or Supabase via `DATABASE_URL` change) |

`DATABASE_URL` is read by `src/shared/infra/db/client.ts` and `drizzle.config.ts`.

## Test isolation

Integration tests use a real Postgres (Railway CI environment in CI, testcontainers locally). Each Vitest worker writes to its own schema (`pg_temp_<workerId>`) so parallel runs don't collide. Wiring lands in task #6.
